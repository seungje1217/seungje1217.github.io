컴퓨터 관련 전공학생들이면 다들 한번씩은 들어봤을거다. 우선 정의를 먼저 한번 보자.
<br/>

> 프로세스(Process) : 운영체제로부터 자원을 할당받은 작업의 단위


> 스레드(Thread) : 프로세스가 할당받은 자원을 이용하는 실행 흐름의 단위

<br/>
이해하기 쉽게 비유를 들어보자.
<br/><br/>
프로세스는 맥도날드 가게(독립된 임대 공간)  
<br/>
스레드는 맥도날드의 직원들(재고, 계산대, 공간 공유)
<br/>
<br/>
정의와 비유는 이 정도로 정리하고,
이제 실제로 무엇이 다르고 왜 문제가 발생하는지 구조적으로 살펴보자.
<br/><br/>

---

### 프로그램

<br/>
프로세스와 스레드에 대해 자세히 설명을 하기 위해서는 먼저 프로그램에 대해 알아야한다.
<br/>
프로그램을 모르는 사람이 어딨어,,할 수 있지만 막상 프로그램에 대해 설명해봐라 하면 제대로 답하는 사람이 거의 없을거다.
<br/><br/>
우선 프로그램의 정의은 파일이 저장 장치에 저장되어 있지만 메모리에는 올라가 있지 않은 정적인 상태를 말한다.
<br/><br/>
운영체제가 메모리 공간을 할당해 줘야 프로그램이 실행되는데, 아직 메모리 공간을 할당해 주지 않아서 메모리에 올라가 있지 않다 하는 거다. 또 그렇기에 실행되지 않아서 정적(static)인 상태가 되는 거고..
<br/><br/>
벌써 어렵다. 쉽게 설명하면 아까 프로세스가 가게라고 하고 스레드가 직원들이라 했으면 프로그램은 그 가게의 설계도라고 생각하면 된다.
<br/><br/>
결론은 프로그램은 아직 실행되지 않은 (설계도니까 아직 만들지 않은) 상태인거다.
우리가 쓰는 윈도우의 *.exe 파일이나 맥의 *.dmg 파일처럼 사용자가 눌러 실행하기 전의 파일을 말한다. **코드 덩어리**라고 생각하면 편하다.
<br/>

---

### 프로세스

<br/>
그럼 정적인 프로그램을 실행시켜 동적인 상태, 즉 작업 중인 상태로 만들면 뭐라할까?
이러한 상태의 프로그램을 바로 프로세스라고 한다.

![image](/assets/images/os/process.png)


위 사진처럼 작업 관리자를 열어보면 지금 실행중인 프로세스들을 볼 수 있다.
<br/><br/>
모든 프로그램들은 운영체제가 메모리 공간을 할당해야 실행될 수 있는데, 프로그램이 실행하게 되면 파일은 컴퓨터 메모리에 올라가게 되고, 운영체제로부터 시스템 자원(CPU)을 할당받아 프로그램 코드를 실행시켜 서비스를 이용할 수 있게 되는거다.
<br/><br/>
쉽게 설명하면 프로그램은 가게의 설계도인데, 실행을 하는 순간 운영체제가 가게를 하나 열어주는 거다. 메모리 안에 공간을 내어주고 코드, 변수, 함수들을 그 공간에 배치한다. 이게 바로 메모리에 올라간다는 말이다.
<br/><br/>
똑같은 애플리케이션을 실행하면 프로세스(동적), 실행하지 않으면 프로그램(정적)인거다.
<br/>

### 스레드

<br/>
스레드는 프로세스가 할당받은 자원을 이용하는 실행 흐름의 단위다.
<br/><br/>
과거에는 하나의 프로세스가 하나의 작업을 처음부터 끝까지 처리하는 방식이 일반적이었다.
하지만 프로그램이 점점 복잡해지면서 한 번에 여러 작업을 동시에 처리해야 하는 상황이 생겼다.
<br/><br/>
예를 들어, 화면을 렌더링하면서 사용자 입력을 받고 네트워크 통신을 처리해야 한다.
<br/><br/>
하나의 실행 흐름으로 이 모든 일을 처리하면 중간에 대기 시간이 발생하고, 프로그램은 멈춘 것처럼 보이게 된다.
<br/><br/>
운영체제는 프로세스마다 독립된 메모리 공간을 부여하는데,
이 덕분에 안정성은 확보되지만 프로세스를 여러 개 생성하면 메모리와 자원 사용 비용이 커진다.
<br/><br/>
그래서 등장한 개념이 바로 스레드다.
<br/><br/>
스레드는 하나의 프로세스 안에서 자원을 공유하면서 동시에 여러 작업을 수행할 수 있게 해준다.
<br/><br/>
예를 들어 맥도날드 가게를 생각해보자.
<br/><br/>
손님이 들어오면 주문을 받고, 햄버거를 만들고, 계산을 하고, 음식을 전달해야 한다.
<br/><br/>
만약 직원이 한 명뿐이라면 어떻게 될까?
<br/><br/>
첫 번째 손님의 주문을 받고<br/>
햄버거를 만들고<br/>
감자튀김을 튀기고<br/>
계산까지 끝낸 다음에야<br/>
다음 손님의 주문을 받을 수 있다.
<br/><br/>
이렇게 되면 가게는 금방 줄이 길어지고 손님들은 기다리게 된다.
<br/><br/>
하지만 직원이 여러 명이라면 상황이 달라진다.<br/>
같은 가게 안에서 여러 작업이 동시에 진행되기 때문에 훨씬 빠르게 일을 처리할 수 있기때문이다.
<br/>

> 즉, 프로세스가 프로그램의 실행 환경이라면 스레드는 그 안에서 실제로 일을 수행하는 실행 흐름이다.

### 프로세스의 자원 구조

<br/>
지금까지 프로세스와 스레드의 개념에 대해 알아보았다. 
<br/><br/>
그렇다면 여기서 한 가지 질문이 생긴다.
<br/><br/>
프로세스가 운영체제로부터 할당받는 자원은 구체적으로 무엇일까?
<br/><br/>
프로세스는 단순히 “실행 중인 프로그램”이라는 개념에 그치지 않는다.
실제로는 운영체제로부터 메모리 공간을 할당받아 그 안에서 코드와 데이터를 실행한다.
<br/><br/>
즉, 프로세스는 자신만의 메모리 구조를 가지고 동작한다.
<br/><br/>
아래의 사진은 하나의 프로세스가 메모리를 어떻게 사용하고 있는지 보여주는 구조이다.
<br/>

![image](/assets/images/os/memory.png)

- 코드 영역(Code / Text)
    - 프로그래머가 작성한 프로그램 함수들의 코드가 CPU가 해석 가능한 기계어 형태로 저장

- 데이터 영역(Data)
    - 코드가 실행되면서 사용하는 전역 변수나 각종 데이터들의 모임

- 스택 영역(Stack)
    -  지역 변수 같은 호출한 함수가 종료되면 돌아올 임시 자료를 저장하는 독립적 공간. 함수의 호출과 함께 할당되며, 호출이 완료되면 소멸
    - 스택 영역 초과 시 stack overflow 발생

- 힙 영역
    - 생성자, 인스턴스와 같은 동적으로 할당되는 데이터들을 위해 존재하는 공간
    - 사용자에 의해 메모리 공간이 동적으로 할당, 해제

<br/>
위 사진에서 스택과 힙 영역의 위아래로 화살표는 스택 영역과 힙 영역은 프로세스가 실행되는 동안 크기가 늘어났다 줄어들기도 하는 동적 영역을 표현한 것이다.
<br/><br/>
그럼 화살표가 없는 코드와 데이터 영역은? 선언할 때 그 크기가 결정되는 정적 영역이다.
<br/><br/>
즉 하나의 프로세스는 이런 메모리 구조를 가지며 실행된다.
<br/><br/>
그렇다면 스레드는 이 메모리 구조에서 어떤 자원을 공유하고 어떤 자원을 독립적으로 사용할까?
<br/><br/>

### 스레드의 자원 구조

<br/>
하나의 프로세스 안에서 실행되는 여러 스레드는 이 자원을 어떻게 사용할까?

![image](/assets/images/os/memory2.png)


스레드는 같은 프로세스 안에서 동작하기 때문에 대부분의 자원을 공유한다.
<br/><br/>
프로세스의 4가지 메모리 영역 중  Code, Data, Heap은 프로세스에 속한 모든 스레드가 함께 사용한다.
<br/><br/>
이 영역들은 프로세스 단위로 할당되는 자원이기 때문에 스레드마다 따로 존재하지 않는다.
즉 하나의 스레드가 힙 영역에 생성한 객체나 데이터를 다른 스레드가 가져와 읽거나 수정할 수 있다.
<br/><br/>
하지만 스레드는 각각 독립적인 실행 흐름을 가지기 때문에 Stack 영역은 스레드마다 별도로 할당된다.
<br/><br/>
스택에는 함수 호출 정보, 지역 변수, 실행 흐름 정보 등이 저장되는데 만약 여러 스레드가 하나의 스택을 공유하게 되면 실행 흐름이 서로 섞여 프로그램이 정상적으로 동작할 수 없기 때문이다.
<br/><br/>
하지만 이러한 구조는 장점만 있는 것은 아니다.
<br/><br/>
스레드가 대부분의 메모리를 공유한다는 것은 성능 면에서는 유리하지만 안정성 면에서는 취약할 수 있다.
<br/><br/>
여기서 프로세스와 스레드의 중요한 차이를 하나 더 알 수 있다.
<br/><br/>
만약 어떤 프로세스 하나에서 오류가 발생하여 강제로 종료된다면 다른 프로세스에는 어떤 영향이 있을까?<br/><br/>
공유하고 있는 파일을 손상시키는 경우가 아니라면 보통 다른 프로세스에는 영향을 주지 않는다.
프로세스는 각각 독립된 메모리 공간을 사용하기 때문이다.
<br/><br/>
기본적으로 각 프로세스는 메모리 상에서 서로 다른 주소 공간(Address Space)에서 실행된다.
따라서 한 프로세스는 다른 프로세스의 변수나 자료구조에 직접 접근할 수 없다.
<br/><br/>
하지만 스레드는 상황이 다르다.
<br/><br/>
스레드는 같은 프로세스 안에서 Code, Data, Heap 영역을 공유한다.
그래서 하나의 스레드에서 치명적인 오류가 발생하면 같은 프로세스에 속한 다른 스레드들 역시 함께 종료된다.
<br/><br/>
코딩을 하다 보면 Segmentation Fault 같은 오류로 프로그램 전체가 종료되는 경험을 해본 적이 있을 것이다.
이는 하나의 실행 흐름에서 발생한 오류가 프로세스 전체에 영향을 주기 때문이다.
<br/><br/>
프로세스를 하나의 프로그램이라고 보면, 스레드는 그 안에서 실행되는 각각의 함수 흐름과 비슷한 역할을 한다.
따라서 하나의 실행 흐름에서 치명적인 오류가 발생하면 같은 프로세스 안에서 동작하는 모든 실행 흐름이 함께 중단되는 것이다.
<br/><br/>

### 프로세스의 자원 공유

<br/>
지금까지 설명을 보면 한 가지 의문이 생길 수 있다.
<br/><br/>
프로세스는 서로 독립된 메모리 공간을 사용한다고 했는데
그렇다면 프로세스끼리는 정말 아무 자원도 공유할 수 없는 것일까?
<br/><br/>
기본적으로는 그렇다.
각 프로세스는 서로 다른 주소 공간에서 실행되기 때문에
한 프로세스가 다른 프로세스의 변수나 메모리에 직접 접근하는 것은 불가능하다.
<br/><br/>
하지만 실제 운영체제에서는 여러 프로그램이 서로 데이터를 주고받아야 하는 상황이 많다.
예를 들어 웹 브라우저와 다운로드 프로그램, 또는 서버와 데이터베이스 같은 프로그램들은 서로 데이터를 교환해야 한다.
<br/><br/>
이러한 문제를 해결하기 위해 운영체제에서는
프로세스 간 통신(IPC, Inter Process Communication) 방법을 제공한다.
<br/><br/>
대표적인 IPC 방식에는 Pipe,	Message Queue, Shared Memory, Socket이 있다.
<br/><br/>
이러한 메커니즘을 통해 서로 독립된 프로세스들도 데이터를 교환하며 협력할 수 있다.
<br/><br/>

### 정리

<br/>
지금까지 프로세스와 스레드의 개념과 메모리 구조에 대해 살펴보았다.
<br/><br/>
프로세스는 운영체제로부터 자원을 할당받아 실행되는 프로그램의 단위이며
각각 독립된 메모리 공간을 사용한다.
<br/><br/>
반면 스레드는 하나의 프로세스 안에서 실행되는 흐름의 단위로
Code, Data, Heap 영역을 공유하면서 동작한다.
<br/><br/>
이러한 구조 덕분에 스레드는 빠르고 효율적으로 여러 작업을 동시에 처리할 수 있지만
자원을 공유하기 때문에 안정성 측면에서는 주의가 필요하다.
<br/><br/>
다음 글에서는 이러한 특징을 바탕으로
멀티 프로세스와 멀티 스레드의 차이에 대해 살펴보자.
