### 멀티 프로세스

<br/>

멀티 프로세스(Multi Process)는 하나의 프로그램을 여러 개의 **프로세스로 나누어 동시에 실행하는 방식**이다.
<br/><br/>
앞에서 설명했듯이 프로세스는 각각 독립된 메모리 공간(Address Space)을 가진다.
따라서 여러 프로세스가 동시에 실행되더라도 서로의 메모리에 직접 접근할 수 없다.
<br/><br/>
예를 들어 웹 브라우저를 생각해보자.
<br/><br/>
크롬(Chrome)은 각 탭을 하나의 프로세스로 실행한다.
어떤 탭에서 오류가 발생해도 다른 탭이 종료되지 않는 이유는 바로 이 때문이다.
<br/><br/>
이해하기 쉽게 맥도날드를 비유로 들어보자 <br/>
프로세스가 맥도날드 가게 하나라면, 멀티 프로세스는 맥도날드 지점을 여러 개 여는 것과 비슷하다.
<br/><br/>
강남점, 홍대점, 신촌점처럼<br/>
각 가게는 완전히 독립된 공간에서 운영된다.
<br/><br/>
만약 홍대점에서 문제가 생겨 문을 닫더라도   강남점이나 신촌점은 정상적으로 운영된다.
<br/><br/>
이처럼 멀티 프로세스는 **각 프로세스가 서로 독립적으로 동작하기 때문에 안정성이 높다.**
<br/><br/>
하지만 멀티 프로세스 구조가 항상 좋은 것만은 아니다.
<br/><br/>
프로세스는 서로 완전히 독립된 메모리 공간을 사용하기 때문에
프로세스를 새로 생성하는 데에는 비교적 많은 비용이 들어간다.
<br/>
또한 여러 프로세스를 동시에 실행하려면
운영체제가 CPU를 번갈아가며 할당해 주어야 한다.
<br/><br/>
여기서 등장하는 개념이 바로 Context Switching이다.
<br/><br/>
Context Switching은 CPU가 한 프로세스를 실행하다가 다른 프로세스로 작업을 바꾸는 과정을 말한다.
<br/><br/>
조금 더 쉽게 이해하기 위해 다시 맥도날드 예시를 들어보자.
<br/><br/>
아까 멀티 프로세스를 여러 개의 맥도날드 지점이라고 설명했다.
이번에는 직원 한 명이 여러 손님을 상대하는 상황을 생각해보자.
<br/><br/>
손님 A 주문 받기 → 잠깐 멈춤
<br/>
손님 B 주문 받기 → 잠깐 멈춤
<br/>
손님 C 계산 → 다시 A 햄버거 만들기
<br/><br/>
이렇게 직원이 계속 일을 바꿔가며 처리하면 겉으로 보기에는 여러 손님을 동시에 응대하는 것처럼 보인다.
<br/><br/>
하지만 일을 바꿀 때마다 직원은 이전 주문이 어디까지 진행됐는지 기억해야 하고, 다음 주문 내용을 확인해야 하고, 필요한 재료를 다시 확인해야 한다.
<br/><br/>
이처럼 작업을 바꿀 때마다 준비 과정이 필요한 것이 바로 Context Switching이다.
<br/><br/>
운영체제에서도 비슷하다.
<br/><br/>
CPU가 한 프로세스를 실행하다가 다른 프로세스로 바뀔 때는 CPU 레지스터 상태 저장, 프로그램 실행 위치 저장, 다음 프로세스 상태 불러오기와 같은 작업이 필요하다.
<br/><br/>
이 과정이 바로 Context Switching 비용이다.
<br/><br/>
그래서 멀티 프로세스는 안정성이 높은 대신 이러한 전환 비용이 발생한다는 특징이 있다.
<br/>

> 멀티 프로세스 vs 멀티 프로세서

발음이 비슷해서 둘이 그냥 억양 차이라고 생각할 수 있는데 엄연히 다르다.<br/>
프로세스(process)는 프로그램의 실행 상태를 말하고, 프로세서(processer)는 CPU 코어를 일컫는다.<br/>
따라서 멀티 프로세스는 하나의 프로그램에서 여러 개의 프로세스를 실행하는 것을 의미하고, 멀티 프로세서는 여러 개의 CPU 코어가 하나의 시스템에서 동시에 실행되는 것을 의미한다.

<br/>

> 개발자의 관점에서 본 Multi Process

이론적인 설명은 충분히 했으니 이제 개발자의 관점에서 한 번 생각해보자.
<br/><br/>
“그래서 멀티 프로세스가 우리가 코딩하는 데 실제로 도움이 되는 걸까?”
<br/><br/>
ㅇㅇ당연하다.
특히 CPU 연산이 많은 작업에서는 멀티 프로세스가 큰 도움이 된다.
<br/><br/>
예를 들어 Python에서는 multiprocessing이라는 모듈을 이용해 새로운 프로세스를 생성할 수 있다.
<br/><br/>
코드를 한번 살펴보자.

```
import multiprocessing

def hello():
    print("프로세스 실행")

if __name__ == "__main__":
    p1 = multiprocessing.Process(target=hello)
    p2 = multiprocessing.Process(target=hello)

    p1.start()
    p2.start()

    p1.join()
    p2.join()
```

이 코드는 hello()라는 함수를 실행하는 두 개의 프로세스를 생성한다.
<br/><br/>
start()를 호출하면 새로운 프로세스가 생성되면서 함수가 실행되고,
join()은 해당 프로세스가 끝날 때까지 기다리는 역할을 한다.
<br/><br/>
즉 하나의 프로그램 안에서 여러 개의 프로세스가 동시에 실행되는 구조가 되는 것이다.
<br/><br/>
실제 개발에서는 대량의 데이터 처리, 이미지 처리, 머신러닝 학습, 수치 계산과 같은 CPU 연산 작업과 같은 상황에서 멀티 프로세스를 사용한다.
<br/><br/>
예를 들어 이미지 1000장을 처리해야 한다고 가정해보자.
<br/><br/>
프로세스를 하나만 사용하면 이미지를 하나씩 순서대로 처리해야 한다.
<br/><br/>
하지만 멀티 프로세스를 사용하면 
<br/>프로세스 1 → 이미지 1~250 처리
<br/>프로세스 2 → 이미지 251~500 처리
<br/>프로세스 3 → 이미지 501~750 처리
<br/>프로세스 4 → 이미지 751~1000 처리
<br/><br/>
이처럼 하나의 큰 작업을 여러 개의 프로세스로 나누어 처리할 수 있다.
<br/><br/>
이렇게 되면 각 프로세스가 서로 다른 CPU 코어에서 동시에 작업을 수행할 수 있기 때문에 전체 처리 속도를 크게 향상시킬 수 있다.
<br/><br/>
즉 멀티 프로세스는 단순히 프로그램을 여러 개 실행하는 개념이 아니라, 큰 작업을 여러 개의 작은 작업으로 나누어 병렬로 처리하기 위한 방법이라고 볼 수 있다.
<br/><br/>
특히 데이터 처리, 영상 처리, 머신러닝 학습처럼 CPU 연산이 많은 작업에서는 이러한 구조가 프로그램의 성능을 크게 향상시키는 중요한 요소가 된다.
<br/><br/>

### 멀티 스레드

<br/>

멀티 스레드(Multi Thread)는 하나의 프로세스 안에서 여러 개의 스레드를 동시에 실행하는 방식이다.
앞에서 설명했듯이 스레드는 프로세스가 할당받은 자원을 이용하는 실행 흐름의 단위이다.
<br/><br/>
멀티 프로세스가 여러 개의 가게를 운영하는 구조였다면
멀티 스레드는 가게는 하나지만 직원이 여러 명인 구조라고 생각하면 된다.
<br/><br/>
맥도날드 예시로 다시 생각해보자.
<br/><br/>
직원이 한 명뿐이라면
<br/><br/>
손님 A 주문 받기
→ 햄버거 만들기
→ 감자튀김 튀기기
→ 계산
→ 손님 B 주문 받기
<br/><br/>
이렇게 모든 일을 순서대로 처리해야 한다.
<br/><br/>
하지만 직원이 여러 명이라면 상황이 달라진다.
<br/><br/>
직원 1 → 주문 받기
직원 2 → 햄버거 만들기
직원 3 → 계산
<br/><br/>
이렇게 같은 가게 안에서 여러 작업이 동시에 진행될 수 있다.
<br/><br/>
이것이 바로 멀티 스레드 구조이다.
<br/><br/>
멀티 스레드는 하나의 프로세스 안에서 실행되기 때문에
Code, Data, Heap 영역을 서로 공유한다.
<br/><br/>
그래서 데이터를 주고받는 과정이 빠르고 프로세스를 새로 생성하는 것보다 훨씬 가볍다.
<br/>

> 멀티 스레드의 특징

	•	스레드 생성 비용이 작다
	•	메모리 사용량이 적다
	•	데이터 공유가 쉽다
	•	프로그램 응답성이 좋아진다

예를 들어 웹 서버를 생각해보자.
<br/><br/>
여러 사용자가 동시에 접속하면 각 요청을 하나의 스레드가 처리할 수 있다.
<br/><br/>
요청 1 → Thread 1<br/>
요청 2 → Thread 2<br/>
요청 3 → Thread 3<br/>
<br/><br/>
이렇게 여러 요청을 동시에 처리할 수 있기 때문에 웹 서버의 처리 성능이 크게 향상된다.
<br/><br/>
하지만 멀티 스레드 구조에도 단점이 존재한다.
<br/><br/>
스레드는 같은 프로세스 안에서 메모리를 공유하기 때문에
여러 스레드가 동시에 같은 데이터를 수정하면 문제가 발생할 수 있다.
<br/><br/>
이때 등장하는 개념이 바로 Race Condition(경쟁 상태)이다.
<br/><br/>

### Race Condition

<br/>

Race Condition은 여러 실행 흐름이 같은 데이터를 동시에 수정하려 할 때 발생하는 문제를 말한다.
<br/><br/>
이름 그대로 누가 먼저 실행되느냐에 따라 결과가 달라지는 상황이다.
<br/><br/>
다시 맥도날드 예시로 생각해보자.
<br/><br/>
직원 두 명이 같은 주문서를 보고 있다고 가정하자.
현재 주문서에는 이렇게 적혀 있다.
<br/><br/>
햄버거 재고 : 10개
<br/><br/>
손님 두 명이 동시에 햄버거를 주문한다.
<br/><br/>
직원 A → 재고 확인 → 10개<br/>
직원 B → 재고 확인 → 10개
<br/><br/>
직원 A → 1개 판매 → 재고 9개<br/>
직원 B → 1개 판매 → 재고 9개
<br/><br/>
하지만 실제로는 햄버거가 2개 팔렸기 때문에 재고는 8개가 되어야 한다.
<br/><br/>
이처럼 여러 실행 흐름이 동시에 같은 데이터를 수정하면서
데이터가 꼬이는 문제가 발생할 수 있다.
<br/><br/>
이것이 바로 Race Condition이다.
<br/><br/>
이러한 문제를 해결하기 위해 멀티 스레드 환경에서는 동기화(Synchronization) 기법을 사용한다.
<br/><br/>
대표적인 방법으로는 Mutex, Semaphore, Lock등이 존재한다.
<br/><br/>
이러한 기법들은 한 번에 하나의 스레드만 자원에 접근하도록 제한하여
데이터 충돌을 방지하는 역할을 한다. 이에 대해서는 추후 자세히 설명을 해보겠다.
<br/><br/>

### 정리

<br/>
지금까지 멀티 프로세스와 멀티 스레드의 개념을 살펴보았다.
<br/><br/>
멀티 프로세스는 프로세스마다 독립된 메모리 공간을 사용하기 때문에 안정성이 높지만
프로세스 생성 비용과 Context Switching 비용이 크다.
<br/><br/>
반면 멀티 스레드는 같은 프로세스 안에서 메모리를 공유하기 때문에
더 빠르고 효율적으로 작업을 처리할 수 있지만
Race Condition과 같은 동기화 문제가 발생할 수 있다.
<br/><br/>
따라서 실제 시스템에서는 프로그램의 특성과 목적에 따라
멀티 프로세스와 멀티 스레드를 적절히 선택하거나
두 방식을 함께 사용하는 구조를 사용한다.
<br/><br/>
우리가 아무 생각 없이 누르는 버튼 하나 뒤에는, 수많은 프로세스와 스레드가 동시에 움직이고 있다.
